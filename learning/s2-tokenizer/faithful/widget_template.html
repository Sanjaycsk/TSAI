chapter 1 · The bug that scored zero — shown live

One Hindi letter is several bytes: भ = E0 A4 AD. v1's BPE tore those 3 bytes across two tokens — like a photo ripped in half, each half unreadable alone. The grader reads every token alone, threw both halves away, and भारत disappeared. v2's atoms are whole characters, so no token can ever hold a fraction of a letter. chapter 2 · The new rules — faithful units

Fertility's denominator is no longer "words". A faithful unit is one contiguous run of letters/digits like this or one single visible punctuation mark , — so URLs, brackets and number separators all count. Watch the grader's regex [\p{L}\p{M}\p{N}]+|[^\s\p{L}\p{M}\p{N}] carve a sentence:

fertility(lang) = tokens ÷ faithful units  ·  score = 1000 ÷ (max − min)  ·  hindi penalty = exp(max(0, hi/1.2 − 1)). Long tokens can span several units, so good fertility sits below 1 — the game is no longer "compress hard", it is "compress equally".

What the 0 taught me

My byte-level round-trip was mathematically perfect — concatenate all bytes, decode once. But the decode contract is part of the tokenizer, and the grader's contract says every token must stand alone as text. A tokenizer isn't correct in the abstract; it's correct against a contract.

Calibrate the checker, not just the code

My round-trip test "failed" on ″ → ′′ … and so did the reference solution, identically — that's NFKC, the accepted behavior. Running the published solution through my own verifier told me where the real bar is.

Compression is no longer the goal

Old formula: push fertility down. New formula: make four fertilities equal — 1000/(max−min). A brilliant English tokenizer that leaves Kannada behind scores worse than a modest, fair one. That's the actual multilingual problem frontier labs fight.
