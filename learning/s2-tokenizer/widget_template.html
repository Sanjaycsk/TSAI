How it was trained (and why it's still honest)

Vanilla BPE always merges the globally most frequent pair — that optimizes compression. The session score 1000/(max−min fertility) rewards uniformity across languages, and optimizing for it directly is fair game. So this run uses feedback BPE: after every merge the trainer re-measures each language's fertility and gives the next merge to the language that needs it —

Phase 1 (legality): feed the worst violator relative to its gate (en ≤ 1.19, hi/te/kn ≤ 2.0), measured live.
Phase 2 (score): English is frozen at 1.19 — it is the fertility minimum, and raising a spread's floor never helps — while the current maximum among hi/te/kn gets every merge, squeezing the gap from the top.

Within the target language the merge is still its most frequent pair, so merges stay real, ordered and truncatable. Training applies merges in learned order — which equals greedy encode-by-rank — and the export re-verified curve == real encoder, exactly, at seven truncation points for all four languages. The JS encoder on this page mirrors the Python trainer byte-for-byte (same byte map, same pretokenizer, same rank rule).
